# Robo Sweep Agent Guide

Robo Sweep is a Vite-powered browser game about a robot vacuum cleaning a cozy living room. The first screen is the game itself, rendered on a full-window canvas from `src/main.js` and `src/game/Game.js`. The game is meant to feel good on a tablet in landscape, with no reading required for the player, so changes should preserve quick startup, large touch targets, clear visual feedback, and forgiving interactions.

The main game loop lives in `src/game/Game.js`. It owns the canvas timing, input routing, y-sorted drawing, sock basket persistence, dock reminders, dog and mess events, and coordination between entities. The robot behavior is in `src/game/entities/Robot.js`, the dock maintenance and service UI are in `src/game/entities/Dock.js`, floor dirt and toys are in `src/game/entities/DirtSystem.js`, and room layout plus furniture tap zones are in `src/game/world/Room.js`. Surprise actions are registered from `src/game/actions/DefaultActions.js` through `src/game/actions/ActionRegistry.js`.

Static artwork and audio that ship with the game live under `public/assets/`. Generated source artwork and prompt history live under `art/`, especially `art/raw/`; these are useful for regenerating or editing assets, but the game itself loads from `public/assets/`. The sprite loader in `src/game/core/AssetLoader.js` treats every sprite as optional and the game falls back to procedural drawing when art is missing. Spoken voice clips are loaded by `src/game/core/Voice.js`, and synthesized sound effects are generated by `src/game/core/SoundEngine.js`.

Every shipped sprite and voice clip must be declared in `src/game/core/assetManifest.js`. This file is the source of truth used by both the browser loaders and the build validator. When adding a sprite, put the final game-ready file under `public/assets/sprites/`, add a key to `SPRITE_MANIFEST`, and reference that key from game code with `assets.get('key')` or `assets.getTinted('key', tint)`. When adding a voice line, put the final `.wav` under `public/assets/voice/`, add its key to `VOICE_LINES`, and if it is generated by the local voice pipeline also add the same key to `scripts/gen_voice.py`. Do not leave extra game asset files in `public/assets/` without adding them to the manifest; the build should fail if the public folder and manifest drift apart.

The standard asset check is `npm run check:assets`. It verifies that every manifest entry has a real file, every public sprite or voice file is represented in the manifest, literal `assets.get(...)` and `say(...)` references in source code point at declared keys, and `scripts/gen_voice.py` agrees with the voice manifest. The normal `npm run build` runs that source check, builds with Vite, and then validates the copied `dist/assets` folder. The GitHub Pages workflow runs the same build plus a stricter post-build check requiring `dist/version.json`, so a missing asset should stop CI before `gh-pages` is updated.

For local development, run `npm install` once and then `npm run dev`. The dev server defaults to port `5173`, or uses `PORT` if supplied. Vite also provides a small local `/api/socks` endpoint from `vite.config.js`; it writes `.sock-stash.json` so multiple browsers on the local network can share the laundry basket state. That file is runtime state and should not be committed.

The public site is https://peteallen.github.io/robotgame/. GitHub Pages is configured to publish from the `gh-pages` branch at `/`, not from the GitHub Actions artifact deployment mode. The source branch remains `main`. On every push to `main`, `.github/workflows/pages.yml` installs dependencies with `npm ci`, builds with `GITHUB_PAGES=true` and `VITE_ASSET_VERSION=${{ github.sha }}`, writes `dist/version.json`, force-pushes the built `dist` contents to `gh-pages`, and then explicitly requests a Pages build through the GitHub API. This branch-based flow replaced the artifact deployment flow because GitHub accepted the artifact but repeatedly failed the final Pages deploy step.

Caching is intentionally handled in the app. `src/game/core/assetUrl.js` appends the build SHA to versioned public assets, so sprites and voice clips are requested with URLs like `assets/sprites/dock.png?v=<commit-sha>`. `src/game/core/VersionWatcher.js` checks `version.json` on a timer and when a hidden tab becomes visible; if a newer build is published, it reloads the page with a small query parameter so the browser asks GitHub Pages for fresh HTML. This matters because GitHub Pages can serve cached HTML and assets for several minutes.

Before pushing, run `GITHUB_PAGES=true VITE_ASSET_VERSION=test npm run build` when deployment behavior or public asset paths changed. For ordinary gameplay-only edits, `npm run build` is usually enough, and it still includes the asset checks. Do not commit `dist/`, `node_modules/`, `.sock-stash.json`, local secrets, `.DS_Store`, or generated Python cache files. Be careful around existing uncommitted changes; they are usually active work from the user, so work with them instead of reverting them.

When changing deploy behavior, verify all three layers: the workflow run should succeed, `gh-pages` should move to a new commit, and `gh api repos/peteallen/robotgame/pages/builds/latest` should report `status: built` for that commit. To confirm what the public site is serving, fetch the live HTML and inspect the hashed bundle path, then fetch that bundle or a specific asset directly from `https://peteallen.github.io/robotgame/`.
