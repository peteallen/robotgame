# Robo Sweep Agent Guide

Robo Sweep is a Vite-powered browser game about a robot vacuum cleaning a cozy living room. The first screen is the game itself, rendered on a full-window canvas from `src/main.js` and `src/game/Game.js`. The game is meant to feel good on a tablet in landscape, with no reading required for the player, so changes should preserve quick startup, large touch targets, clear visual feedback, and forgiving interactions.

When documenting or reporting work on this project, write in clear conversational paragraphs instead of terse shorthand. Assume the reader may not know the codebase yet, and give enough surrounding context for file names, systems, and commands to make sense. Use sub-agents when they would make the work safer or faster, especially for parallel investigation or for keeping a long implementation from crowding out important context.

The main game loop lives in `src/game/Game.js`. It owns the canvas timing, input routing (including the press-and-drag rescue of a trapped robot), y-sorted drawing, sock basket persistence, and a watchdog pipeline that drives everything time-based: the poop-smear-mop chain, the all-clean win party, equipment trips to the dock, the trapped-robot and dog-chase timers, and dock service reminders. The robot behavior is in `src/game/entities/Robot.js`, the dock maintenance and service UI are in `src/game/entities/Dock.js`, floor dirt and toys are in `src/game/entities/DirtSystem.js`, the corgi is in `src/game/entities/Dog.js`, and room layout plus furniture tap zones are in `src/game/world/Room.js`. Surprise actions live in themed modules under `src/game/actions/` (`celebrations.js`, `stunts.js`, `chores.js`, `dockTrips.js`, `trapped.js`, shared `helpers.js`) and are registered by `src/game/actions/index.js` through `src/game/actions/ActionRegistry.js`; that index has a short recipe comment for adding a new action.

A few gameplay invariants matter when changing behavior. Debris never appears on its own: every dirt item is player-caused (floor taps, plant shakes, toy launches, sock drags, dog taps), which is what makes the all-clean victory party — lap around the room, then dock, service, and stay parked until new dirt spawns — meaningful. `DirtSystem.spawn` is the single choke point that re-arms the win (`game.roomDirty`) and wakes a parked robot. The robot always smears a fresh poop pile, mop pads or not; pads only let it skip the pad-install dock trip before mopping. A docked robot refuses to leave (and blinks its status light red) while the dock has any unserviced alert. The robot's battery gauge is drawn on the robot itself, not in the HUD, and there is deliberately no score/star system — the HUD is just the dust/mop pill, the mode picker, and the sound toggle.

Static artwork and audio that ship with the game live under `public/assets/`. Generated source artwork and prompt history live under `art/`, especially `art/raw/`; these are useful for regenerating or editing assets, but the game itself loads from `public/assets/`. The sprite loader in `src/game/core/AssetLoader.js` treats every sprite as optional and the game falls back to procedural drawing when art is missing. Spoken voice clips are loaded by `src/game/core/Voice.js`, recorded sound-effect clips (the corgi barks) by `src/game/core/Sfx.js` with synth fallbacks, and synthesized effects are generated live by `src/game/core/SoundEngine.js`. All generation pipelines and their models are documented in the README "Models" table — that table is the canonical registry; keep script defaults and the table in sync. Voice lines come from `scripts/gen_voice.py` and must pass the transcription QA loop (`scripts/voice_qa_loop.sh`); sound effects come from `scripts/gen_sfx.py` (ElevenLabs; key from `ELEVENLABS_API_KEY` or `~/.codex/.env`).

Every shipped sprite, voice clip, and sound effect must be declared in `src/game/core/assetManifest.js`. This file is the source of truth used by both the browser loaders and the build validator. When adding a sprite, put the final game-ready file under `public/assets/sprites/`, add a key to `SPRITE_MANIFEST`, and reference that key from game code with `assets.get('key')` or `assets.getTinted('key', tint)`. When adding a voice line, put the final `.wav` under `public/assets/voice/`, add its key to `VOICE_LINES`, and add the same key to `scripts/gen_voice.py`. When adding a sound effect, put the final `.mp3` under `public/assets/sfx/`, add its key to `SFX_CLIPS`, and add the same key to `scripts/gen_sfx.py`. Do not leave extra game asset files in `public/assets/` without adding them to the manifest; the build should fail if the public folder and manifest drift apart.

The standard asset check is `npm run check:assets`. It verifies that every manifest entry has a real file, every public sprite/voice/sfx file is represented in the manifest, literal `assets.get(...)`, `say(...)`, and `sfx.play(...)` references in source code point at declared keys, and that `scripts/gen_voice.py` and `scripts/gen_sfx.py` agree with their manifests. The normal `npm run build` runs that source check, builds with Vite, and then validates the copied `dist/assets` folder. The GitHub Pages workflow runs the same build plus a stricter post-build check requiring `dist/version.json`, so a missing asset should stop CI before `gh-pages` is updated.

For local development, run `npm install` once and then `npm run dev`. The dev server defaults to port `5173`, or uses `PORT` if supplied. Vite also provides a small local `/api/socks` endpoint from `vite.config.js`; it writes `.sock-stash.json` so multiple browsers on the local network can share the laundry basket state. That file is runtime state and should not be committed.

The public site is https://peteallen.github.io/robotgame/. GitHub Pages is configured to publish from the `gh-pages` branch at `/`, not from the GitHub Actions artifact deployment mode. The source branch remains `main`. On every push to `main`, `.github/workflows/pages.yml` installs dependencies with `npm ci`, builds with `GITHUB_PAGES=true` and `VITE_ASSET_VERSION=${{ github.sha }}`, writes `dist/version.json`, force-pushes the built `dist` contents to `gh-pages`, and then explicitly requests a Pages build through the GitHub API. This branch-based flow replaced the artifact deployment flow because GitHub accepted the artifact but repeatedly failed the final Pages deploy step.

Caching is intentionally handled in the app. `src/game/core/assetUrl.js` appends the build SHA to versioned public assets, so sprites and voice clips are requested with URLs like `assets/sprites/dock.png?v=<commit-sha>`. `src/game/core/VersionWatcher.js` checks `version.json` on a timer and when a hidden tab becomes visible; if a newer build is published, it reloads the page with a small query parameter so the browser asks GitHub Pages for fresh HTML. This matters because GitHub Pages can serve cached HTML and assets for several minutes.

Before pushing, run `GITHUB_PAGES=true VITE_ASSET_VERSION=test npm run build` when deployment behavior or public asset paths changed. For ordinary gameplay-only edits, `npm run build` is usually enough, and it still includes the asset checks. Do not commit `dist/`, `node_modules/`, `.sock-stash.json`, local secrets, `.DS_Store`, or generated Python cache files. Be careful around existing uncommitted changes; they are usually active work from the user, so work with them instead of reverting them.

When changing deploy behavior, verify all three layers: the workflow run should succeed, `gh-pages` should move to a new commit, and `gh api repos/peteallen/robotgame/pages/builds/latest` should report `status: built` for that commit. To confirm what the public site is serving, fetch the live HTML and inspect the hashed bundle path, then fetch that bundle or a specific asset directly from `https://peteallen.github.io/robotgame/`.
